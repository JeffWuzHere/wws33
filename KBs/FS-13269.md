# Knowledge Base Entry: Wholesale Order Form Category Filter Width CSS Fix + WordPress User Enumeration via REST API

## 🎯 Executive Summary
**The Issue (CSS):** The category filter dropdown in the Wholesale Order Form was too narrow, causing category names to wrap; a customer also could not find how to widen the "Select Category" element via plugin settings.
**The Fix (CSS):** Custom CSS added via Appearance > Customize > Additional CSS widens the `.category-filter` and related elements to prevent wrapping. The agent applied the CSS directly to the staging site and the customer confirmed it worked.

**The Issue (Security):** After sharing admin credentials with support, the customer observed brute-force attempts on their admin username and suspected a breach.
**The Fix (Security):** The root cause was WordPress user enumeration via the REST API (`/wp/v2/users`) and `?author=` query parameters — not a support-side breach. Adding code to `functions.php` to restrict the REST API user endpoint and block author queries resolved the enumeration.

---

## 🛠 Final Resolution (The "2nd Me" Guide)
> [!TIP]
> **Use this solution if a customer asks about:** category filter too narrow, select category wrapping, order form header element width, category dropdown CSS, custom CSS order form layout
>
> **Steps to solve (Category Filter Width):**
> 1. Go to Appearance > Customize > Additional CSS in the WordPress admin.
> 2. Add the following CSS (adjust `425px` to suit the store's needs):
> ```css
> /* Select Category field — wider */
> .category-filter.header-footer-element {
>     width: 425px !important;
>     max-width: 425px !important;
> }
> .category-filter .category-filter-box {
>     width: 425px !important;
> }
> .category-filter .wwof-category-filter {
>     width: 100% !important;
> }
> /* Dropdown panel to match */
> .ant-tree-select-dropdown {
>     min-width: 425px !important;
>     width: 425px !important;
> }
> /* Adjust search input if needed */
> .search-input.header-footer-element {
>     width: 220px !important;
>     max-width: 220px !important;
> }
> .search-input .search-input-group {
>     width: 220px !important;
> }
> ```
> 3. **Note:** Do NOT add `flex-wrap: nowrap` to `.wwof-row.ant-row` — this breaks the mobile layout.
> 4. Clear site caches and confirm on the frontend.

> [!TIP]
> **Use this solution if a customer asks about:** WordPress admin username brute force after sharing credentials, user enumeration REST API, wp-json users exposed, block author query WordPress, stop username harvesting
>
> **Steps to solve (User Enumeration):**
> 1. Explain that WordPress treats usernames as public information and the REST API endpoint `/wp/v2/users` exposes them by default. This is a known WordPress behavior: https://make.wordpress.org/core/handbook/testing/reporting-security-vulnerabilities/#why-are-disclosures-of-usernames-or-user-ids-not-a-security-issue
> 2. Add the following to the active theme's `functions.php` or via a code manager plugin (e.g., WPCode):
> ```php
> // Restrict REST API user enumeration
> add_filter('rest_endpoints', function($endpoints) {
>     if (!current_user_can('list_users')) {
>         unset($endpoints['/wp/v2/users']);
>         unset($endpoints['/wp/v2/users/(?P<id>[\d]+)']);
>     }
>     return $endpoints;
> });
>
> // Block ?author= queries
> add_action('template_redirect', function() {
>     if (isset($_GET['author']) && !is_admin()) {
>         wp_redirect(home_url());
>         exit;
>     }
> });
>
> // Redirect author archive pages
> add_action('template_redirect', function() {
>     if (is_author()) {
>         wp_redirect(home_url());
>         exit;
>     }
> });
> ```
> 3. The customer confirmed this closed the enumeration loophole without breaking the site.

## 💡 Internal Insights (If applicable)
- The `flex-wrap: nowrap` CSS rule on `.wwof-row.ant-row` causes the order form header to break on mobile — do not include it when applying this fix.
- The "favourites" (British English) vs "favorites" (US English) text issue was escalated as a bug for the next WWOF release; Loco Translate is the correct workaround until then.
- WordPress user enumeration is a platform-level issue, not a Wholesale Suite bug. The REST API and `?author=` parameter are the two primary vectors. The `functions.php` snippet above is a reliable mitigation.
