# Knowledge Base Entry: Wholesale Order Form WPML String Translations Broken After Update to v3.0.9+

## 🎯 Executive Summary
**The Issue:** After updating WooCommerce Wholesale Order Form to v3.0.9+, WPML string translations stopped working — all order form strings reverted to English regardless of the active language.
**The Fix:** The translation system was changed from a legacy workaround to the WordPress-standard JSON format (`wp_set_script_translations`); a hotfix was released that added the required `.json` translation files, restoring translations. Customers on affected versions should update to the latest WWOF release.

---

## 🛠 Final Resolution (The "2nd Me" Guide)
> [!TIP]
> **Use this solution if a customer asks about:** WPML translations not working order form, order form strings showing in English, WPML string scan finds nothing, translations missing after update, WWOF 3.0.9 WPML broken, JSON translation files, wp_set_script_translations
>
> **Steps to solve:**
> 1. Confirm the customer is running WWOF v3.0.9 or later (the translation system changed in this version).
> 2. Explain that WWOF moved from a legacy WPML workaround to the WordPress-standard JSON translation format (`wp_set_script_translations`). WPML's "Scan Plugin" tool will no longer surface WWOF strings as it did before — translations are now handled via `.po`/`.mo`/`.json` files.
> 3. Direct the customer to update to the latest available WWOF version. The hotfix that adds bundled language files (DE, IT, FR, ES, and others) was released shortly after v3.0.9.
> 4. If translations are still missing after updating, point them to the Loco Translate workflow for generating custom `.json` files: https://wholesalesuiteplugin.com/kb/how-to-translate-wholesale-suite-plugins/#2-loco-translate
> 5. If they cannot update yet, advise them to remain on WWOF v3.0.8 as a temporary workaround — the old WPML string approach works on that version.

## 📝 Sanitized Interaction History

**[CUSTOMER]:**
After installing the latest version of WooCommerce Wholesale Order Form, WPML string translations stopped working on the order form. All strings show in English even when on a German-language page. Scanning the plugin with WPML's string translation tool returns no results. Reverting to WWOF v3.0.8 restores the translations.

**[SUPPORT_AGENT]:**
The translation system was changed in WWOF v3.0.9. The previous implementation used a workaround because WPML did not natively support JavaScript translations at the time. WordPress now requires JavaScript strings to be translated via JSON files (see: https://developer.wordpress.org/reference/functions/wp_set_script_translations/). We have moved to that standard format. Bundled translation files for the requested languages will be included in a hotfix. In the meantime, please remain on v3.0.8.

**[CUSTOMER]:**
Installed the latest minor update released today — it looks fixed. We can close this ticket.

## 💡 Internal Insights (If applicable)
- The root change was `wp_set_script_translations` adoption. WPML's "Scan Plugin" string discovery does not pick up strings loaded this way — customers using WPML must use Loco Translate to generate the `.json` sidecar file alongside their `.po`/`.mo` files.
- Customers needing languages not bundled by default should use Loco Translate to generate the required JSON: https://wholesalesuiteplugin.com/kb/how-to-translate-wholesale-suite-plugins/#2-loco-translate
- Reference WPML errata: https://wpml.org/errata/translating-strings-from-javascript-files-using-wp_set_script_translations/
